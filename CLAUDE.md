# CLAUDE.md

Internal WooCommerce accounting/reporting system (not tax accounting). The full authoritative spec is **README.md** — read it before planning or implementing. If this file and README ever disagree, README wins.

## Stack & Commands

- Laravel 12, MySQL 8 (`woocommerce_accounting`), Pest, Pint. Domain code lives in `app/Domain/{Accounting,Orders,Channels,Costing,Products,Receivables,Expenses,Reports,Sync}`.
- **Frontend is Blade + Alpine.js only — the React/Inertia migration is COMPLETE (2026-07-12).** Every screen (dashboard, orders, products, tools, settings, warehouse, notifications, reports, review, fast-forms, users, auth, system-settings, components) renders through Laravel Blade with the TailAdmin design system (`resources/views/**`). Interactivity is plain GET/POST forms with full page reloads — no fetch/AJAX; Alpine only for cosmetic client-side bits (dropdowns, modals, column toggles).
  - **React and Inertia must NOT be used for any new feature or page.** The legacy React/Inertia code is deleted and deprecated — do not restore it, do not reintroduce `Inertia::render`, and do not add a `resources/js/pages/**` tree. Controllers return `view(...)`; there is no Inertia middleware, no Inertia root view, and no SSR build. `resources/js/` contains only the Alpine/TailAdmin bundle (`resources/js/tailadmin/**`); the single Vite entry pair is `resources/css/tailadmin.css` + `resources/js/tailadmin/app.js`.
  - New UI must be built from the existing Blade component system (`resources/views/components/**` — see the component showcase below), not from scratch and not from a new library.
  - **ApexCharts is the official charting library** (wired via `resources/js/tailadmin/app.js`); new report/dashboard screens follow the `DashboardController`/`pages.dashboard.*` pattern.
  - **No new UI/frontend framework or chart library may be introduced without a separate architectural decision.** Document gaps instead of installing around them.
- **Component showcase** (`/components`, `ShowcaseController`, admin-only): internal reference gallery of every reusable Blade + Alpine component, rendered via its real implementation with safe mock data (never DB fixtures). The registry is `config/showcase.php`; each component has a **stable machine id** (`card-01`, `table-01`, `chart-01`, …) and a Persian reference phrase («کارت شماره ۱»). **IDs are permanent** — never renumber an existing component (ids are the source of truth, not array position); a new component takes the next free number in its category, and numbering is independent per category. **Every new reusable Blade component should be added to the showcase.** The showcase uses only the supported Blade + Alpine architecture (no React/Inertia) and **no new UI/chart library may be introduced without a separate architectural decision** — document gaps instead. Charts reuse the real dashboard widgets, each with its own unique ApexCharts element id (never duplicate ids on one page). Registry-driven: the `کامپوننت‌ها` sidebar group (admin-only, built in `MenuHelper`) and the category pages both read `config/showcase.php`, so they can't drift.
- **Reporting Design System** (Phase 2, 2026-07-12) — the foundation every dashboard/report/admin screen must build on. **Extend it; never write page-specific UI that re-invents it.**
  - **Design tokens** live in `resources/css/tailadmin.css` → `@theme`, as *semantic aliases over* the TailAdmin primitives (never new palettes): financial (`--color-profit|loss|income|expense|trend-*`), the 9 statuses (`--color-status-*` + `-bg`), surfaces, `--radius-card|control|badge`, `--shadow-card|dropdown|dialog`, chart/row heights, motion. Tailwind 4 compiles `text-profit` to `color: var(--color-profit)`, and the tokens are **re-declared under `.dark`**, so semantic utilities are theme-aware with **no `dark:` variant needed**. Tokens consumed only via `var()` (chart/row heights, durations) live in a plain `:root` block — Tailwind tree-shakes `@theme` vars it can't see used as a class.
  - **Status is resolved in exactly one place:** `App\Support\Design\StatusPresenter` (label + token, safe fallback for unknown values). Render it with `<x-ui.status>`. A status always shows **dot + text label** — never colour alone (WCAG).
  - **Numbers/money: `<x-tables.num>` is mandatory.** It sets `dir="ltr"` (so digits and the minus sign read correctly) **and** `text-right` + tabular figures. This supersedes the old center-align rule: numeric columns are **right-aligned with tabular figures** so amounts stack by place value (۱٬۲۵۰٬۰۰۰ over ۶۴۰٬۰۰۰) — the financial-reporting standard, and in RTL the right edge *is* the reading start, so header and value share one edge. Never hand-write `<td dir="ltr">` again: that flips the cell's base direction, making `text-align: start` resolve to *left* while the header stays right — the bug that broke every financial column.
  - **Charts: `<x-charts.chart preset="…">` only — never a hard-coded element id.** The id is generated per instance and the config travels as `data-chart` JSON; `resources/js/tailadmin/charts/` holds the preset registry (line, area, bar, bar-horizontal, bar-stacked, donut, pie, mixed, waterfall, heatmap, radial, kpi-trend, comparison) and one initializer that also **re-renders every chart on theme toggle** (ApexCharts does not follow `.dark` on its own). A preset can appear any number of times on a page. Adding a chart type = one entry in `presets.js`, no new JS wiring. **ApexCharts remains the only chart library.**
  - **KPI cards:** `<x-kpi.card>` (value, trend with arrow **and** sign, sparkline, progress, status). **Data states:** `<x-states.state variant="empty|no-results|error|permission|loading|skeleton">` — every data surface shows one instead of a blank area.
  - **Blade trap:** never write a component tag inside a `//` PHP comment **or a JS comment** — Blade's tag compiler scans raw text and will compile it (it broke every KPI page in Phase 2 with `withAttributes() on null`). `{{-- --}}` comments are stripped first and are safe.
- **Phase 3 (2026-07-12) completed the design system.** New rules:
  - **Server-driven table state lives in the URL.** `App\Support\Design\TableQuery` owns search, **multi-column sort** (`?sort=-total,date`), per-column search (`?c[col]=`), page size (`?per_page`), filters, and the active-filter chips. Every URL it builds **preserves the other params** (sorting never drops a filter; paging never drops the sort) — so a report link always shows the sender what the recipient sees. Sorting/filtering/paging are **GET round-trips; never add a client-side data store.** Pass a `TableQuery` into `<x-tables.pro-table :query>` to get chips + page-size + the no-results state for free.
    - **Every list page now uses it** (orders, products, customers, deposits, bank-account transactions). A page **declares** which columns sort (`'sort' => '<key>'` in its column array) and `pro-table` derives the header link, arrow and multi-sort priority badge from the `TableQuery`. **Never hand-build a sort URL in a view again** — that is how a sort click used to silently drop the active filter. Controller: `->tap(fn ($q) => $query->apply($q))->paginate($query->perPage())`.
    - The URL contract is `?search=` (not `?q=`), `?sort=-total,date` (leading `-` = desc), `?per_page=`. Ctrl/⌘-click a header to add a second sort column.
  - **`<x-tables.num>` is mandatory for every number** (money, id, quantity, percentage), and **`<x-tables.ltr>` for every LTR identifier** (email, IBAN, card/phone number, SKU, invoice no, Jalali date — same bug, not a number). Types: `int|decimal|money|toman|rial|percent`; `signed` adds profit/loss colour **and** an explicit +/- sign; `digits="fa"` for Persian codepoints (default `latin` keeps figures tabular); `zero` distinguishes a real zero from an unavailable `—`. It owns `dir="ltr"` + `text-right` + tabular figures together — the alignment fix must never be re-implemented per page.
  - **The primitive owns the figure's COLOUR too — pick a `tone`, never pass a `text-*` class.** `tone` = `default|muted|subtle|positive|negative` (`brand` on `ltr`); the colour sits on the figure span, so any colour class on the `<td>` is a silent no-op. Why it exists: `body` sets no text colour, so a figure that inherited colour from its cell fell back to the browser default (**pure black**) whenever a caller forgot to pass one — illegible in dark mode, and heavier than its neighbours in light. **Each tone inverts with the background** (darker shade on light, lighter on dark); pairing them the other way round (`text-gray-400 dark:text-gray-500`) fails WCAG AA at *both* ends. All tones are AA-verified in both themes; `--color-profit` is success-**700** (not -600, which is only 3.7:1 on white) — it is text, not just a chart fill.
  - **Charts:** 16 presets in `resources/js/tailadmin/charts/presets.js` — line, area, bar, bar-horizontal, bar-stacked, donut, pie, mixed, waterfall, heatmap, radial, kpi-trend, comparison, **cumulative-target, revenue-orders, pnl-bridge**. Adding a type = one entry there. Never a hard-coded element id, never page-specific chart JS.
  - **Composition over duplication — the core rule.** The design system deliberately has *few* primitives with *many* variants. Before creating a component, check whether one of these already covers the shape: `<x-financial.summary>` (every label→number→total widget: profit, cash flow, receivables, payables, loans, cheques, tax, monthly closing), `<x-financial.breakdown>` (chart + share table), `<x-financial.bridge>` (waterfall), `<x-widgets.ranked-list>` (every "top N"/leaderboard), `<x-widgets.timeline>` (activity, notifications, events), `<x-widgets.quick-stats>`, `<x-widgets.target-progress>`, `<x-kpi.card>` / `<x-kpi.compact>` / `<x-kpi.comparison>`, `<x-todo.list>`, `<x-filters.{panel,chips,quick}>`, `<x-nav.{tabs,pagination,section-header}>`, `<x-states.state>`. **Do not add a near-duplicate component; add a variant/prop.**
  - **Components take data via props. No DB queries and no business logic in Blade views** — the controller or a service prepares the data (dates arrive already Jalali-formatted).
  - **TODO components have no backend.** `<x-todo.list>` toggles completion in Alpine only; the item contract is documented at the top of the file so a backend can attach later without changing the markup.
  - **Every new component MUST be registered in `config/showcase.php` immediately after it is created** — with a stable id, `tech` (English identifier), Persian name, description, variants, source path, usage, and known limitations, plus a live preview in `components/showcase/example.blade.php`. IDs are permanent and per-category; adding a category is allowed, renumbering is not. The catalog at `/components` is the reference future requests use to name a component ("use table-06", "add chart-08").
- **Data tables** (orders, products, invoices, any list view): use `<x-tables.pro-table>` (toolbar) over `<x-tables.data-table>` (grid) — never fork a second table. Rounded bordered card, search + filters + column-visibility + **density modes** (compact/default/comfortable) in the toolbar, sortable headers with arrow icons, sticky header, checkbox row selection (header indeterminate state) + bulk-action bar, status badges (dot + label), per-row ⋮ action menu, footer totals + pagination. Server-driven: sorting/filtering/pagination are GET round-trips; Alpine only holds cosmetic state (visibility, density, selection). Column `align` is **independent** of `key` — a column must be alignable without opting into column-visibility. RTL, IRANSansX, Persian digits. The ⋮ row-action menu must open toward the side it has room on — verify per layout. Approved by user 2026-07-11; alignment rule revised 2026-07-12 (see above).
- Money is integer Toman (BIGINT); all journal writes go through `App\Domain\Accounting\Services\JournalPoster` (balanced, idempotent, period-lock aware — never insert journal rows directly).
- Roles (Spatie permission): `admin`, `accountant`, `warehouse`, `partner_viewer`. Convention in `routes/web.php`: reads open to `admin|accountant|warehouse`; financial mutations (shipping/packaging cost, recalc, cost/wholesale mapping) `admin|accountant` only; users/tools/settings/warehouse-config `admin` only; reports readable by `admin|accountant|partner_viewer`, finalize `admin` only.

## Data Sources

- Real data comes only from the mirrored hub: `behdashtik-hub-main/`. Never query the production WooCommerce site directly.
- `dev.behdashtik.ir` is for discovery and safe tests only. Dev data must never create real accounting records or reports.

## Non-Negotiable Rules

- Currency: Toman everywhere. Timezone: `Asia/Tehran`. User-facing dates: Jalali/Shamsi.
- Store display name is configurable; the app name is not tied to the store name.
- Current phase never writes to WooCommerce (no prices, stock, wholesale prices, or orders).
- Wholesale prices are internal only.
- Sensitive financial data (purchase cost, profit, margins, payroll, banks, loans, cheques, partner reports) must never leak via public APIs, webhooks, logs, exports, or WooCommerce. General logs must not contain sensitive values — use protected audit logs.
- Missing purchase cost or missing Cost Mapping is never treated as zero — such orders go to review queues.
- Every financial number must be explainable, traceable, and auditable. Use reversal/voiding and adjustments, never silent edits or deletion.
- Product-page cost/wholesale entry (`ProductController::storeCost`/`setWholesale`) is profit-discovery data only — it feeds `CostResolver`/`ProfitEngine` so order profit/loss can be computed, and must never create a `Party`, `PurchaseInvoice`, or `JournalEntry`. Real purchases (a real supplier, a real accounts-payable entry) are recorded exclusively through `/new-buy-order` (`PurchaseInvoiceController` + `PurchaseInvoiceService`). Don't blur this boundary again.

## Architecture Principles

- Prefer configurable mappings over hard-coded business rules (statuses, channels, source mapping, cost centers, thresholds).
- Sales channels are dynamic and data-driven — never hard-code a channel list. Unknown sources are stored, queued for review, and reported with a fallback label; they must never break processing.
- Sync must be idempotent: external ID mapping, retry, dead-letter queue, correlation IDs, no duplicate orders or accounting entries.
- Tests are mandatory for all financial logic (see README §24 for the required list).
